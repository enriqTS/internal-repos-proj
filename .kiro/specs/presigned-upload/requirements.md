# Requirements Document

## Introduction

This feature replaces the current multipart upload flow (frontend → API Gateway → Lambda) with a presigned URL approach. The current architecture forces the entire project folder through API Gateway, which has a hard 10MB payload limit. Large repos with build artifacts frequently exceed this limit, resulting in 413 errors. The new flow uploads files directly from the browser to S3 via presigned URLs, then triggers Lambda processing via a lightweight metadata-only API call. This removes the payload ceiling entirely and decouples file transfer from request processing.

## Glossary

- **Staging_Bucket**: A dedicated S3 bucket (or prefix within the existing bucket) where raw uploaded files are temporarily stored before Lambda processing
- **Presigned_URL**: A time-limited, pre-authenticated URL generated by the backend that allows the frontend to upload directly to S3 without passing through API Gateway
- **Upload_Session**: A server-generated identifier (UUID) that groups the presigned URL, the staged file, and the subsequent processing request together
- **Initiate_Endpoint**: The API Gateway endpoint that validates metadata, creates the Upload_Session, and returns a Presigned_URL for S3 upload
- **Finalize_Endpoint**: The API Gateway endpoint called after S3 upload completes, which triggers Lambda to process the staged file into a Project_Entry
- **Client_Zip**: A zip archive created in the browser (using JSZip or equivalent) from the selected folder contents, applying client-side DENY_LIST filtering before compression
- **Processing_Lambda**: The Lambda function that downloads the Client_Zip from the Staging_Bucket, extracts it, applies server-side filtering (DENY_LIST + .gitignore), generates the artifact, readme, metadata, and regenerates the Search_Index
- **Lifecycle_Policy**: An S3 lifecycle rule that automatically deletes incomplete/abandoned uploads from the Staging_Bucket after a configurable expiration period

## Requirements

### Requirement 1: Upload Initiation

**User Story:** As an employee, I want to request an upload session before sending files, so that the system can validate my metadata upfront and give me a direct path to upload to S3.

#### Acceptance Criteria

1. THE Initiate_Endpoint SHALL accept a POST request with a JSON body containing project name (required), tags (optional), and readme content (optional), and SHALL validate these fields using the same rules as the current upload flow (name format, tag count/length limits, readme length limit)
2. WHEN validation succeeds, THE Initiate_Endpoint SHALL generate a unique Upload_Session ID and a Presigned_URL for S3 PUT to the Staging_Bucket at the path `staging/{session-id}/upload.zip`, with an expiration time of 15 minutes
3. WHEN validation succeeds, THE Initiate_Endpoint SHALL return a JSON response containing the Upload_Session ID, the Presigned_URL, and the expiration timestamp
4. IF validation fails, THEN THE Initiate_Endpoint SHALL return a 400 error with a message indicating which fields are invalid, without generating a Presigned_URL or Upload_Session
5. IF a project with the same name already exists in S3, THEN THE Initiate_Endpoint SHALL return a 409 error indicating the project name is already taken
6. THE Initiate_Endpoint SHALL require a valid API key in the x-api-key header, consistent with the existing authentication mechanism

### Requirement 2: Client-Side Zip and Direct S3 Upload

**User Story:** As an employee, I want my browser to zip my project files and upload directly to S3, so that I am not constrained by API Gateway's payload size limit.

#### Acceptance Criteria

1. WHEN the user submits the upload form, THE Frontend SHALL apply client-side DENY_LIST filtering to the selected files, excluding files matching denied patterns before zipping
2. AFTER filtering, THE Frontend SHALL create a Client_Zip in-memory using a JavaScript zip library, preserving directory structure relative to the project root
3. AFTER creating the Client_Zip, THE Frontend SHALL upload the zip file directly to S3 using the Presigned_URL received from the Initiate_Endpoint via an HTTP PUT request
4. WHEN the S3 upload completes successfully (HTTP 200), THE Frontend SHALL proceed to call the Finalize_Endpoint
5. IF the S3 upload fails, THEN THE Frontend SHALL display an error message indicating the file upload failed and allow the user to retry
6. THE Frontend SHALL display upload progress to the user during the S3 PUT operation using XMLHttpRequest or fetch progress events
7. IF no files remain after client-side DENY_LIST filtering, THEN THE Frontend SHALL display an error message indicating no uploadable files were found and SHALL NOT call the Initiate_Endpoint

### Requirement 3: Upload Finalization and Processing

**User Story:** As an employee, I want the system to process my uploaded files after they reach S3, so that my project appears in the search index with proper filtering applied.

#### Acceptance Criteria

1. THE Finalize_Endpoint SHALL accept a POST request with a JSON body containing the Upload_Session ID
2. WHEN the Finalize_Endpoint receives a valid Upload_Session ID, THE Processing_Lambda SHALL download the Client_Zip from the Staging_Bucket at path `staging/{session-id}/upload.zip`
3. THE Processing_Lambda SHALL extract the Client_Zip, apply server-side filtering (DENY_LIST + .gitignore rules), generate artifact.zip, write readme.md and metadata.json, and regenerate the Search_Index — using the same logic as the current Upload_Lambda
4. WHEN processing completes successfully, THE Finalize_Endpoint SHALL return a success response containing the project path and any warnings (e.g., .gitignore parse failures)
5. IF the Upload_Session ID does not correspond to an existing staged file in S3, THEN THE Finalize_Endpoint SHALL return a 404 error indicating the upload session was not found or has expired
6. IF server-side filtering removes all files from the zip, THEN THE Processing_Lambda SHALL return a 400 error indicating no files remain after filtering
7. WHEN processing completes (success or failure), THE Processing_Lambda SHALL delete the staged zip file from the Staging_Bucket
8. THE Finalize_Endpoint SHALL require a valid API key in the x-api-key header

### Requirement 4: Staging Bucket Infrastructure

**User Story:** As a developer, I want staged uploads to be automatically cleaned up, so that abandoned or failed uploads do not accumulate storage costs.

#### Acceptance Criteria

1. THE Infrastructure SHALL provision a Staging_Bucket (or a staging prefix within the existing bucket) with a Lifecycle_Policy that expires objects under the `staging/` prefix after 24 hours
2. THE Staging_Bucket SHALL have CORS configured to allow PUT requests from the frontend origin
3. THE Staging_Bucket SHALL block public access; only presigned URLs and the Processing_Lambda's IAM role SHALL have access to objects
4. THE Processing_Lambda's IAM role SHALL have permissions to GetObject and DeleteObject on the Staging_Bucket's `staging/` prefix
5. THE Initiate_Endpoint's Lambda (or execution role) SHALL have permissions to generate presigned PutObject URLs for the Staging_Bucket's `staging/` prefix

### Requirement 5: Upload Size Limits

**User Story:** As an administrator, I want reasonable size limits on uploads, so that the system is not abused for storing excessively large files.

#### Acceptance Criteria

1. THE Presigned_URL SHALL include a content-length condition limiting the upload to a maximum of 500 MB
2. IF the Client_Zip exceeds 500 MB after client-side filtering, THE Frontend SHALL display an error message indicating the project is too large to upload and SHALL NOT attempt the S3 upload
3. THE Processing_Lambda SHALL enforce the existing 100 MB limit on the final artifact.zip after server-side filtering; IF the final artifact exceeds 100 MB, THEN THE Processing_Lambda SHALL return a 400 error indicating the artifact exceeds the maximum allowed size
