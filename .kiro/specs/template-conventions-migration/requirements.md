# Requirements Document

## Introduction

This feature migrates both chatbot RAG templates (`chatbot-rag-agentcore` and `chatbot-rag-mantle`) from their current dependency management and tooling setup to the upd8 company conventions for Python and Terraform. The migration replaces per-Lambda `requirements.txt` files with a centralized `pyproject.toml` managed by `uv` (Astral), adds `ruff` as formatter and linter, introduces `pytest` test infrastructure, and aligns Terraform configuration with upd8 standards (standardized tags, naming conventions, backend configuration, and file organization). Both templates share the same structural changes — the migration is applied identically to each.

## Glossary

- **Template_Root**: The root directory of a chatbot RAG template (e.g., `templates/chatbot-rag-mantle/`), containing all project scaffold files.
- **UV_Manager**: The `uv` (Astral) Python package manager used for virtual environment creation, dependency resolution, and lock file generation.
- **Pyproject_Config**: The `pyproject.toml` file at Template_Root defining all Python project metadata, dependencies, and tool configurations.
- **UV_Lock**: The `uv.lock` file at Template_Root committed to version control for reproducible dependency resolution.
- **Ruff_Tool**: The `ruff` (Astral) linter and formatter tool configured via the `[tool.ruff]` section in Pyproject_Config.
- **Lambda_Packaging**: The process of producing zip artifacts for Terraform's `archive_file` data source, using `uv export --format requirements-txt` to bridge uv dependencies with Lambda deployment.
- **Standard_Tags**: The set of mandatory Terraform resource tags: `Project`, `Environment`, `ManagedBy = "terraform"`, and `Client`.
- **Resource_Naming_Convention**: The pattern `${var.project_name}-${var.environment}-<function>` used for all AWS resource names.
- **Backend_Config**: The Terraform S3 backend configuration following the pattern `upd8-tfstate-<cliente>` with DynamoDB lock table `upd8-tfstate-lock` and encryption enabled.
- **Shared_Lambda_Layer**: The Lambda Layer at `src/layers/shared/` containing shared utilities (logging config, data models) used by all Lambda functions.

## Requirements

### Requirement 1: Centralized Dependency Management with uv

**User Story:** As a developer working on a chatbot template, I want all Python dependencies managed centrally via `pyproject.toml` and `uv`, so that I have a single source of truth for dependencies with reproducible installs.

#### Acceptance Criteria

1. THE Template_Root SHALL contain a `pyproject.toml` file declaring all Python project dependencies including runtime dependencies (aws-lambda-powertools, boto3, openai) and development dependencies (pytest, hypothesis, ruff) with lower-bound version constraints (using compatible-release `~=` or minimum-version `>=` operators) for each dependency.
2. THE Template_Root SHALL contain a `uv.lock` file committed to version control, generated by UV_Manager, providing reproducible dependency resolution across all environments.
3. WHEN a developer clones the template with only `uv` installed as a prerequisite, THE developer SHALL be able to install all dependencies by running `uv sync` at Template_Root with no other manual configuration, tool installation, or environment variable setup required.
4. THE Pyproject_Config SHALL declare at least two dependency groups — one containing runtime Lambda dependencies and one containing development-only dependencies (testing, linting) — enabling selective export of only the runtime group for Lambda artifact packaging.
5. THE per-Lambda `requirements.txt` files (e.g., `src/ai_caller/requirements.txt`, `src/orchestrator/requirements.txt`) SHALL be removed from the template, replaced by the centralized Pyproject_Config.
6. THE Pyproject_Config SHALL specify `requires-python = ">=3.12"` to match the Lambda runtime target.
7. WHEN Lambda artifacts are built, THE build process SHALL export only the runtime dependency group from Pyproject_Config to produce the dependency set for each Lambda, ensuring development-only dependencies are excluded from deployed artifacts.
8. WHEN Pyproject_Config dependencies are added, removed, or their version constraints are modified, THE UV_Manager SHALL regenerate the `uv.lock` file so that the lock file remains consistent with the declared dependencies before changes are committed to version control.

### Requirement 2: Lambda Packaging with uv export

**User Story:** As a developer deploying the chatbot, I want Lambda packaging to work with the centralized uv dependency management, so that Terraform can still produce zip artifacts without manual pip install steps.

#### Acceptance Criteria

1. THE Template_Root README SHALL document the Lambda packaging process using `uv export --format requirements-txt` to generate a pinned requirements file compatible with Terraform's packaging workflow.
2. WHEN Terraform builds Lambda zip artifacts via `archive_file`, THE packaging process SHALL include third-party dependencies installed from the uv-exported requirements into the Lambda source directory, targeting the Lambda runtime platform (linux, matching the configured function architecture).
3. THE Template_Root SHALL include a build script or Makefile target that automates the packaging of each Lambda function, running `uv export --format requirements-txt` from the project's `pyproject.toml` and installing dependencies into each Lambda function's source directory (co-located with handler code) before `terraform apply`.
4. THE Shared_Lambda_Layer packaging SHALL use the same uv export approach, installing layer dependencies into the `src/layers/shared/python/` directory structure required by the Lambda Layer convention.
5. THE Template_Root README deployment section SHALL replace all `pip install -r requirements.txt -t <target>` instructions with equivalent `uv` commands.
6. IF the build script fails during `uv export` or dependency installation for any Lambda function or layer, THEN THE build process SHALL exit with a non-zero status code and print an error message indicating which packaging step failed, preventing `terraform apply` from proceeding with incomplete artifacts.

### Requirement 3: Ruff Configuration

**User Story:** As a developer writing Python code for the chatbot, I want ruff configured as both formatter and linter, so that code quality is enforced consistently without needing separate tools.

#### Acceptance Criteria

1. THE Pyproject_Config SHALL include a `[tool.ruff]` section with `target-version = "py312"` configuring ruff as the project's formatter and linter.
2. THE Pyproject_Config `[tool.ruff.lint]` section SHALL enable at minimum the rule sets: `E` (pycodestyle errors), `F` (pyflakes), `I` (isort), `UP` (pyupgrade), and `B` (flake8-bugbear).
3. THE Pyproject_Config `[tool.ruff]` section SHALL set `line-length = 120` to accommodate Lambda handler code readability.
4. THE Template_Root `.gitignore` SHALL include `.ruff_cache/` to exclude ruff cache files from version control.
5. THE Template_Root README SHALL document usage of ruff via `uv run ruff format .` for formatting and `uv run ruff check .` for linting.
6. THE template SHALL NOT include any `black`, `flake8`, or `isort` configuration files or dependencies, as ruff replaces all of these tools.
7. THE Pyproject_Config SHALL declare `ruff` as a dev dependency so that `uv run ruff format .` and `uv run ruff check .` resolve without additional manual installation.

### Requirement 4: Type Hints and Code Quality

**User Story:** As a developer reading the chatbot template code, I want all functions to have type hints, so that the code is self-documenting and compatible with static analysis tools.

#### Acceptance Criteria

1. THE Lambda handler functions and all utility functions in the template Python source files (all `.py` files under `src/`) SHALL have type annotations on all parameters and return types, excluding `self` and `cls` parameters which follow standard convention.
2. THE Pyproject_Config `[tool.ruff.lint]` section SHALL include rule `ANN` (flake8-annotations) to enforce type hint presence, and violations SHALL cause the `ruff check` command to exit with a non-zero status code.
3. THE shared models in `src/layers/shared/python/shared/models.py` SHALL use Python dataclasses or TypedDict with full type annotations for all fields.
4. IF type information is unavailable for a value because the third-party library provides no typed stubs or type exports (e.g., boto3 response payloads), THEN THE code SHALL use explicit `Any` type annotation with an inline comment stating the library name and reason the concrete type cannot be determined.
5. WHEN `ruff check` is executed against the template source files, THE linter SHALL report zero `ANN`-category violations for the template to be considered compliant.

### Requirement 5: Pytest Test Infrastructure

**User Story:** As a developer extending the chatbot template, I want a pytest test infrastructure ready to use, so that I can write and run tests from the start.

#### Acceptance Criteria

1. THE Template_Root SHALL contain a `tests/` directory with at minimum one example test file per Lambda function, where each test file demonstrates mocking AWS services using `unittest.mock` (or `pytest` monkeypatch fixtures), setting required environment variables via fixtures, and invoking the handler with a sample event payload.
2. THE Pyproject_Config SHALL include `pytest` and `hypothesis` as development dependencies under a `[dependency-groups]` dev group.
3. THE Pyproject_Config SHALL include a `[tool.pytest.ini_options]` section specifying `testpaths = ["tests"]` and a `pythonpath` entry enabling imports from the Lambda source directory.
4. WHEN a developer runs `uv run pytest` at Template_Root, THE test suite SHALL execute all discovered tests and exit with code 0, requiring no real AWS credentials, no live AWS services, and no pre-set environment variables outside the test fixtures.
5. THE Template_Root README SHALL include a "Testing" section containing: the command to run tests (`uv run pytest`), the directory convention for new test files (`tests/test_<function_name>.py`), and an explanation of the mocking pattern used in the example tests.
6. IF a Lambda handler depends on environment variables at import time, THEN THE corresponding test file SHALL set those variables in a session-scoped or function-scoped fixture that executes before the handler module is imported.

### Requirement 6: Terraform Backend Configuration Alignment

**User Story:** As a platform engineer deploying the template, I want the Terraform backend configured to upd8 standards, so that state management follows company conventions from the start.

#### Acceptance Criteria

1. THE `backend.tf` file in each environment folder (`environment/dev`, `environment/staging`, `environment/prod`) SHALL configure an S3 backend with bucket naming following the pattern `upd8-tfstate-<cliente>`, region set to `us-east-1`, encrypted state, and a DynamoDB lock table named `upd8-tfstate-lock`.
2. THE `backend.tf` SHALL include `encrypt = true` to enforce server-side encryption of the state file.
3. THE `backend.tf` SHALL include `dynamodb_table = "upd8-tfstate-lock"` for state locking to prevent concurrent modifications.
4. THE `backend.tf` SHALL use a `key` path following the pattern `<project>/terraform.tfstate`, where `<project>` is a placeholder the developer replaces with the actual project name before running `terraform init`.
5. THE Template_Root README SHALL include instructions for the developer to replace the `<cliente>` placeholder in the backend bucket name and the `<project>` placeholder in the state key with their actual values before running `terraform init`.
6. THE Template_Root README SHALL document that the S3 bucket (`upd8-tfstate-<cliente>`) and the DynamoDB table (`upd8-tfstate-lock`) must exist before running `terraform init`, and provide the commands or reference to create them if they do not exist.

### Requirement 7: Terraform File Organization

**User Story:** As a platform engineer reading the Terraform code, I want the files organized per upd8 convention, so that I can find configuration in predictable locations.

#### Acceptance Criteria

1. EACH environment folder SHALL contain at minimum the following separate files: `backend.tf` (backend config), `providers.tf` (provider declarations with version constraints), `variables.tf` (variable declarations), `main.tf` (module calls and resources), and `outputs.tf` (output declarations).
2. THE `providers.tf` file SHALL contain a `terraform` block with `required_version` and `required_providers`, plus the `provider "aws"` block, extracted from `main.tf` where they currently reside. It SHALL NOT contain backend configuration.
3. THE `backend.tf` file SHALL contain a `terraform` block with only the `backend "s3"` configuration.
4. THE `main.tf` file SHALL contain only module calls and resource definitions, with no `terraform` block, no `provider` block, and no backend configuration.
5. WHEN the reorganization is complete, THE system SHALL produce a `terraform plan` output reporting zero additions, zero changes, and zero destructions against the current remote state — confirming this is a file reorganization only with no infrastructure behavior change.

### Requirement 8: Standardized Terraform Tags

**User Story:** As a platform engineer, I want all AWS resources tagged with standard upd8 metadata, so that cost allocation, ownership, and management tooling work consistently.

#### Acceptance Criteria

1. THE Terraform configuration SHALL define variables for `project_name` (string, max 64 characters), `environment` (string, restricted to `dev`, `staging`, or `prod`), and `client` (string, max 64 characters) at the environment level, each with a type declaration and description, used to populate Standard_Tags on all resources.
2. THE Terraform configuration SHALL apply the Standard_Tags (`Project`, `Environment`, `ManagedBy = "terraform"`, `Client`) to all taggable AWS resources provisioned by the template (Lambda functions, DynamoDB tables, S3 buckets, SQS queues, API Gateway, IAM roles).
3. THE tags SHALL be defined as `default_tags` in the provider block so that all resources inherit them automatically without requiring per-resource tag blocks.
4. THE `terraform.tfvars.example` SHALL include the `project_name`, `environment`, and `client` variables with non-empty example values (e.g., `project_name = "my-project"`) and a comment on each variable describing its purpose and allowed values.
5. IF `project_name` or `client` is set to an empty string, THEN THE Terraform configuration SHALL fail validation with an error message indicating which variable must not be empty.

### Requirement 9: Resource Naming Convention

**User Story:** As a platform engineer, I want all resource names to follow the upd8 naming pattern, so that resources are easily identifiable by project and environment in the AWS console.

#### Acceptance Criteria

1. THE Terraform configuration SHALL construct resource names using the pattern `${var.project_name}-${var.environment}-<function>` for all named AWS resources (Lambda functions, SQS queues, DynamoDB tables, S3 buckets, IAM roles, API Gateway, and CloudWatch resources), where `<function>` is a lowercase, hyphen-separated descriptor of the resource's purpose.
2. THE `project_name` variable SHALL be declared with type `string`, a description, and a validation rule that enforces lowercase alphanumeric characters and hyphens only, with a maximum length of 20 characters. THE `environment` variable SHALL be declared with type `string`, a description, and a validation rule that restricts values to `dev`, `staging`, or `prod`.
3. THE existing `project_prefix` variable SHALL be removed, and all resource name references previously using that variable SHALL be updated to use the computed pattern `${var.project_name}-${var.environment}-<function>`.
4. THE module interface (variables passed to child modules) SHALL accept both `project_name` and `environment` as separate required variables, and each module SHALL internally compute the compliant name prefix from these two inputs.
5. THE `terraform.tfvars.example` SHALL contain commented examples showing how `project_name = "my-chatbot"` and `environment = "dev"` combine to produce resource names like `my-chatbot-dev-orchestrator`, and SHALL include a note that the combined prefix plus function suffix must not exceed 63 characters to respect AWS naming limits.

### Requirement 10: README Migration

**User Story:** As a developer setting up the chatbot template, I want the README to reflect uv-based workflows, so that I follow the correct setup and deployment commands.

#### Acceptance Criteria

1. THE Template_Root README prerequisites section SHALL list `uv` (Astral) as a required tool instead of `pip`, with a link to the official uv installation page (https://docs.astral.sh/uv/).
2. THE README deployment section SHALL replace all `pip install -r requirements.txt -t <target>` commands with the equivalent uv workflow: `uv sync` for local development, and `uv export --format requirements-txt` combined with the Terraform-driven Lambda packaging step for deployment.
3. THE README SHALL include a "Development Setup" section documenting: `uv sync` for installing dependencies, `uv run ruff format .` for formatting, `uv run ruff check .` for linting, and `uv run pytest` for running tests.
4. THE README SHALL document that `uv.lock` is committed to version control for reproducibility and SHALL NOT be added to `.gitignore`.
5. THE README SHALL remove all references to `pip`, `virtualenv`, or `venv` as setup tools, replacing each with its uv equivalent: `pip install` replaced by `uv sync` or `uv add`, `virtualenv`/`venv` creation replaced by `uv venv`, and `pip freeze` replaced by `uv export`.
6. WHEN the README migration is complete, THE README SHALL contain no remaining occurrences of `pip install`, `pip freeze`, `virtualenv`, `python -m venv`, or `requirements.txt` as user-facing setup instructions.

### Requirement 11: Template .gitignore Updates

**User Story:** As a developer working on the chatbot template, I want the .gitignore to be aligned with the uv/ruff toolchain, so that generated files are properly excluded while important lock files are tracked.

#### Acceptance Criteria

1. THE Template_Root `.gitignore` SHALL include `.venv/` to exclude the uv-managed virtual environment directory.
2. THE Template_Root `.gitignore` SHALL include `.ruff_cache/` to exclude ruff's cache directory.
3. THE Template_Root `.gitignore` SHALL NOT include `uv.lock`, as the lock file is committed to version control for reproducibility.
4. THE Template_Root `.gitignore` SHALL remove entries for `venv/` and `env/` directory patterns that reference the old virtualenv approach, consolidating to `.venv/` only (the default uv creates), while preserving any `.env` dotenv file exclusion pattern that protects secrets from being committed.
5. THE Template_Root `.gitignore` SHALL retain all existing Python bytecode (`__pycache__/`, `*.pyc`), Terraform (`.terraform/`, `*.tfstate*`), build artifact, and OS file exclusions unchanged.

### Requirement 12: Both Templates Consistency

**User Story:** As a developer comparing the two chatbot templates, I want the conventions migration applied identically to both, so that the only difference remains the AI service integration layer.

#### Acceptance Criteria

1. THE `chatbot-rag-agentcore` template SHALL have the same pyproject.toml sections (metadata, scripts, tool.ruff, tool.pytest), uv.lock presence, ruff configuration values, pytest configuration values, Terraform tag definitions, resource naming convention, backend configuration structure, and directory layout as the `chatbot-rag-mantle` template, excluding only template-specific identifier values.
2. THE pyproject.toml for the AgentCore template SHALL differ from the Mantle template only in the `[project.dependencies]` section, where AgentCore declares `boto3` (for Bedrock agent runtime calls) and Mantle declares `openai` (for Bedrock Mantle API calls).
3. THE Terraform module files, variable definitions, output definitions, provider configurations, and resource structures SHALL be identical across both templates, with the only permitted differences being: (a) the presence of the `modules/agentcore/` module in the AgentCore template, (b) the template-specific value in the backend S3 key path, and (c) the default value of the `project_name` variable.
4. THE README for each template SHALL use identical content for the tooling sections (uv setup, ruff usage, pytest running, Terraform deployment) and SHALL differ only in the AI service description section, architecture diagram references, and AI-specific environment variable documentation.
5. WHEN a shared file (any file outside the AI integration layer) is modified in one template, THE same modification SHALL be applied to the corresponding file in the other template within the same commit.
